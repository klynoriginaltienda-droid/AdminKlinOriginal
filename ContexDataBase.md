## 📋 Tablas — Definición Completa

### 1. `marcas`
Catálogo de marcas de calzado.

```sql
create table marcas (
  id        bigint generated by default as identity primary key,
  nombre    text not null unique,   -- "Nike", "Adidas", "New Balance"
  slug      text not null unique,   -- "nike", "adidas", "new-balance"
  logo_url  text                    -- URL de Cloudinary del logo de la marca
);
```

**Reglas de negocio:**
- `slug` debe ser generado automáticamente desde `nombre` (lowercase, sin espacios, con guiones).
- `logo_url` es opcional, pero recomendado para UI.

---

### 2. `categorias`
Tipos de calzado disponibles.

```sql
create table categorias (
  id      bigint generated by default as identity primary key,
  nombre  text not null unique,  -- "Zapatillas", "Botas", "Sandalias", "Mocasines"
  slug    text not null unique   -- "zapatillas", "botas", "sandalias", "mocasines"
);
```

---

### 3. `generos`
Segmentación de mercado por género.

```sql
create table generos (
  id      bigint generated by default as identity primary key,
  nombre  text not null unique  -- "Hombre", "Mujer", "Unisex", "Niño", "Niña"
);
```

---

### 4. `tallas`
Catálogo de tallas disponibles con soporte multisistema.

```sql
create table tallas (
  id      bigint generated by default as identity primary key,
  valor   text not null unique,         -- "38", "39", "40", "41", "42"
  sistema text not null default 'PE'    -- "PE" (Perú), "US", "EU"
);
```

**Reglas de negocio:**
- Por defecto el sistema de tallas es peruano (`PE`).
- `valor` es siempre string para soportar tallas como "XS", "M", "L" en el futuro.

---

### 5. `colores`
Colores disponibles para variantes de productos.

```sql
create table colores (
  id      bigint generated by default as identity primary key,
  nombre  text not null unique,  -- "Negro", "Blanco", "Rojo Carmesí"
  hex     text                   -- "#000000" (para mostrar swatch en UI)
);
```

**Reglas de negocio:**
- `hex` es opcional pero muy recomendado para filtros visuales en frontend.
- Formato hex siempre con `#` incluido y 6 caracteres: `#RRGGBB`.

---

### 6. `productos`
Tabla principal del catálogo. Representa el modelo de calzado (sin color ni talla aún).

```sql
create table productos (
  id            bigint generated by default as identity primary key,
  modelo        text not null,                    -- "Air Max 90", "Stan Smith"
  codigo        text,                             -- "REF-001", "KLY-123" (Código personalizado)
  slug          text not null unique,             -- "nike-air-max-90" (para URLs SEO)
  id_marca      bigint references marcas(id),
  id_categoria  bigint references categorias(id),
  id_genero     bigint references generos(id),
  descripcion   text,                             -- descripción larga del producto
  precio_base   numeric(10,2) not null,           -- precio sin descuento en soles (S/)
  precio_oferta numeric(10,2),                    -- null = sin oferta activa
  estado        text not null default 'activo'
                check (estado in ('activo', 'inactivo', 'agotado', 'borrador')),
  destacado     boolean default false,            -- aparece en sección "Destacados"
  created_at    timestamptz default now(),
  updated_at    timestamptz default now(),
  deleted_at    timestamptz                       -- null = activo | fecha = eliminado (soft delete)
);
```

**Reglas de negocio:**
- Nunca hacer `DELETE` físico en `productos`. Usar soft delete: `update productos set deleted_at = now() where id = ?`.
- `precio_oferta` debe ser siempre menor que `precio_base`. Validar en backend con Zod.
- `slug` se genera concatenando `marca.slug + "-" + modelo` en lowercase sin espacios.
- `estado = 'borrador'` → visible solo para admins, no para clientes.
- `estado = 'agotado'` → visible para clientes pero sin opción de compra.
- `updated_at` se actualiza automáticamente mediante trigger (ver sección de triggers).
- Todo query de frontend debe incluir `where deleted_at is null`.

**Estados del producto:**

| Estado | Visible para clientes | Comprable |
|---|---|---|
| `activo` | ✅ | ✅ |
| `borrador` | ❌ | ❌ |
| `inactivo` | ❌ | ❌ |
| `agotado` | ✅ | ❌ |

---

### 7. `variantes`
Combinación Producto + Color. Cada variante es una unidad de venta distinta con su propio SKU.

```sql
create table variantes (
  id          bigint generated by default as identity primary key,
  id_producto bigint references productos(id) on delete cascade,
  id_color    bigint references colores(id),
  sku         text unique,            -- Código único: "NIK-AM90-BLC-001"
  precio      numeric(10,2),          -- null = hereda precio_base del producto padre
  activo      boolean default true,
  unique (id_producto, id_color)      -- un producto no puede tener 2 variantes del mismo color
);
```

**Reglas de negocio:**
- Un producto puede tener múltiples variantes (una por color).
- Si `precio` es `null`, el frontend usa `productos.precio_base`.
- Si `precio` tiene valor, ese precio tiene prioridad (para ediciones limitadas o colores premium).
- `sku` debe ser único en todo el sistema. Formato sugerido: `{MARCA}-{MODELO_ABREV}-{COLOR_ABREV}-{NUMERO}`.
- Si `activo = false`, la variante no aparece en tienda.

---

### 8. `variantes_tallas`
Stock real del inventario: por variante (producto+color) y por talla. Esta es la tabla de inventario.

```sql
create table variantes_tallas (
  id_variante bigint references variantes(id) on delete cascade,
  id_talla    bigint references tallas(id) on delete cascade,
  stock       int not null default 0 check (stock >= 0),
  primary key (id_variante, id_talla)
);
```

**Reglas de negocio:**
- `stock` nunca puede ser negativo (constraint `check (stock >= 0)`).
- Al realizar una venta, decrementar stock con una transacción atómica.
- Si `stock = 0` en todas las tallas de todas las variantes de un producto, considerar cambiar `estado` a `agotado`.
- Para agregar una nueva talla a una variante, hacer `INSERT` con el stock inicial.

---

### 9. `imagenes`
Imágenes de Cloudinary, asociadas a una variante específica (cada color tiene sus propias fotos).

```sql
create table imagenes (
  id              bigint generated by default as identity primary key,
  id_variante     bigint references variantes(id) on delete cascade,
  cloudinary_id   text not null,    -- public_id de Cloudinary (ej: "calzado/nike-am90-blanco-frente")
  url             text not null,    -- secure_url completa de Cloudinary
  url_thumbnail   text,             -- URL con transformación eager (ej: w_400,f_auto,q_auto)
  width           int,              -- ancho original en píxeles
  height          int,              -- alto original en píxeles
  orden           int default 0,    -- 0 = imagen principal (portada de la variante)
  created_at      timestamptz default now()
);
```

**Reglas de negocio:**
- `cloudinary_id` (public_id) es la clave para aplicar transformaciones dinámicas sin guardar múltiples URLs.
- `orden = 0` es siempre la imagen principal de la variante.
- Al eliminar una imagen del frontend, también eliminar de Cloudinary usando su `cloudinary_id`.
- `url_thumbnail` se genera al momento del upload con transformación eager: `w_400,h_400,c_fill,f_auto,q_auto`.
- Una variante puede tener múltiples imágenes (ej: frente, perfil, suela, detalle).

## ⚙️ Triggers

```sql
-- Actualiza updated_at automáticamente en cada UPDATE de productos
create or replace function set_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;

create trigger trg_productos_updated_at
  before update on productos
  for each row execute function set_updated_at();
```

## 🔍 Queries SQL de Referencia

### Obtener producto completo (para página de detalle)
```sql
select
  p.id,
  p.modelo,
  p.slug,
  p.descripcion,
  p.precio_base,
  p.precio_oferta,
  p.estado,
  p.destacado,
  m.nombre    as marca_nombre,
  m.slug      as marca_slug,
  cat.nombre  as categoria_nombre,
  g.nombre    as genero_nombre,
  c.nombre    as color_nombre,
  c.hex       as color_hex,
  v.id        as variante_id,
  v.sku,
  v.precio    as variante_precio,
  t.valor     as talla,
  t.sistema   as talla_sistema,
  vt.stock,
  i.url       as imagen_url,
  i.url_thumbnail,
  i.cloudinary_id,
  i.orden     as imagen_orden
from productos p
join marcas m       on m.id = p.id_marca
join categorias cat on cat.id = p.id_categoria
join generos g      on g.id = p.id_genero
join variantes v    on v.id_producto = p.id and v.activo = true
join colores c      on c.id = v.id_color
join variantes_tallas vt on vt.id_variante = v.id
join tallas t       on t.id = vt.id_talla
left join imagenes i on i.id_variante = v.id
where p.slug = 'nike-air-max-90'
  and p.deleted_at is null
  and p.estado = 'activo'
order by v.id, i.orden, t.valor::int;
```

### Listar productos con imagen principal (para catálogo/cards)
```sql
select
  p.id,
  p.modelo,
  p.slug,
  p.precio_base,
  p.precio_oferta,
  p.destacado,
  m.nombre  as marca,
  g.nombre  as genero,
  cat.nombre as categoria,
  -- imagen principal de la primera variante activa
  (
    select i.url_thumbnail
    from variantes v2
    join imagenes i on i.id_variante = v2.id and i.orden = 0
    where v2.id_producto = p.id and v2.activo = true
    limit 1
  ) as imagen_principal,
  -- stock total del producto
  (
    select coalesce(sum(vt2.stock), 0)
    from variantes v3
    join variantes_tallas vt2 on vt2.id_variante = v3.id
    where v3.id_producto = p.id
  ) as stock_total
from productos p
join marcas m       on m.id = p.id_marca
join generos g      on g.id = p.id_genero
join categorias cat on cat.id = p.id_categoria
where p.deleted_at is null
  and p.estado = 'activo'
order by p.created_at desc;
```

### Verificar stock de una talla específica antes de añadir al carrito
```sql
select vt.stock
from variantes_tallas vt
join variantes v on v.id = vt.id_variante
where v.id_producto = $1      -- id del producto
  and v.id_color    = $2      -- id del color seleccionado
  and vt.id_talla   = $3      -- id de la talla seleccionada
  and v.activo      = true;
```

### Decrementar stock al confirmar venta (usar en transacción)
```sql
update variantes_tallas
set stock = stock - $1          -- cantidad a descontar
where id_variante = $2
  and id_talla    = $3
  and stock       >= $1;        -- previene stock negativo
```

## 🖼️ Integración con Cloudinary

### Variables de entorno requeridas
```env
CLOUDINARY_CLOUD_NAME=tu_cloud_name
CLOUDINARY_API_KEY=tu_api_key
CLOUDINARY_API_SECRET=tu_api_secret
NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME=tu_cloud_name
```

### Flujo de upload de imagen
1. Frontend envía el archivo al endpoint `/api/imagenes/upload`
2. Backend sube a Cloudinary con la SDK: `cloudinary.uploader.upload(file, options)`
3. Cloudinary devuelve `{ public_id, secure_url, width, height, ... }`
4. Backend guarda en tabla `imagenes`: `cloudinary_id = public_id`, `url = secure_url`
5. `url_thumbnail` se genera con transformación eager en el mismo upload

### Transformaciones de Cloudinary por caso de uso
```typescript
// Al hacer upload, aplicar transformación eager para thumbnail
const uploadOptions = {
  folder: 'calzado/productos',
  eager: [
    { width: 400, height: 400, crop: 'fill', format: 'auto', quality: 'auto' }
  ],
  eager_async: false
}

// Para generar URLs dinámicas desde cloudinary_id (NO guardar estas URLs)
const urlPrincipal  = `https://res.cloudinary.com/${CLOUD}/image/upload/w_800,f_auto,q_auto/${cloudinaryId}`
const urlThumbnail  = `https://res.cloudinary.com/${CLOUD}/image/upload/w_400,h_400,c_fill,f_auto,q_auto/${cloudinaryId}`
const urlBanner     = `https://res.cloudinary.com/${CLOUD}/image/upload/w_1200,f_auto,q_auto/${cloudinaryId}`
```

### Al eliminar una imagen
```typescript
// Eliminar de Cloudinary primero, luego de la BD
await cloudinary.uploader.destroy(imagen.cloudinary_id)
await supabase.from('imagenes').delete().eq('id', imagenId)
```

## ✅ Validaciones Zod para el Backend

```typescript
import { z } from 'zod'

export const createProductoSchema = z.object({
  modelo:        z.string().min(2).max(100),
  slug:          z.string().min(2).max(120).regex(/^[a-z0-9-]+$/),
  id_marca:      z.number().int().positive(),
  id_categoria:  z.number().int().positive(),
  id_genero:     z.number().int().positive(),
  descripcion:   z.string().max(1000).optional(),
  precio_base:   z.number().positive().multipleOf(0.01),
  precio_oferta: z.number().positive().multipleOf(0.01).optional(),
  estado:        z.enum(['activo', 'inactivo', 'agotado', 'borrador']).default('borrador'),
  destacado:     z.boolean().default(false),
}).refine(
  (data) => !data.precio_oferta || data.precio_oferta < data.precio_base,
  { message: 'precio_oferta debe ser menor que precio_base', path: ['precio_oferta'] }
)

export const createVarianteSchema = z.object({
  id_producto: z.number().int().positive(),
  id_color:    z.number().int().positive(),
  sku:         z.string().max(50).optional(),
  precio:      z.number().positive().multipleOf(0.01).optional(),
  activo:      z.boolean().default(true),
})

export const updateStockSchema = z.object({
  id_talla: z.number().int().positive(),
  stock:    z.number().int().min(0),
})

export const createImagenSchema = z.object({
  id_variante:   z.number().int().positive(),
  cloudinary_id: z.string().min(1),
  url:           z.string().url(),
  url_thumbnail: z.string().url().optional(),
  width:         z.number().int().positive().optional(),
  height:        z.number().int().positive().optional(),
  orden:         z.number().int().min(0).default(0),
})
```

## ⚠️ Reglas Críticas para el Agente

1. **Nunca hacer DELETE físico en `productos`** → Siempre soft delete con `deleted_at = now()`.
2. **Todo query de `productos` debe filtrar** `where deleted_at is null` (excepto panel de admin).
3. **Stock nunca puede quedar negativo** → Usar `check (stock >= 0)` en BD + validación en Zod + query con `AND stock >= cantidad`.
4. **`precio_oferta` siempre debe ser < `precio_base`** → Validar con Zod `.refine()`.
5. **Siempre guardar `cloudinary_id`** junto con la URL → Es necesario para transformaciones y para eliminar de Cloudinary.
6. **Las imágenes se eliminan en dos pasos**: primero de Cloudinary, luego de la BD. Nunca solo de la BD.
7. **`updated_at`** se actualiza automáticamente por trigger, no incluirlo en los `UPDATE` del backend.
8. **El precio efectivo de una variante** = `variante.precio ?? producto.precio_base`. Aplicar esta lógica en todos los cálculos de precio.
9. **`slug` es único y se usa como identificador público** de productos en las URLs. Nunca exponer `id` en URLs públicas de productos.
10. **Variantes inactivas** (`activo = false`) no deben aparecer en ningún endpoint público de tienda.
